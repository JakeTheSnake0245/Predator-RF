What we can leverage
Good news — SDR++ already ships with sdrpp_server (a binary IQ/FFT/VFO streaming protocol over TCP, with zstd compression) and a matching client source module. That gives us the spectrum-streaming half for free — we'd wrap it with our auth layer and a per-peer session, rather than inventing a new wire format. Three decisions that shape the design
Auth model — API key is fine, but where does it live?

Option A (simpler): One API key per node. Same key opens all access (view + control). You set it once in Settings; peers paste it once when adding you. Easy, mirrors how home automation tools work.
Option B (slightly better): Two keys per node — a view-only key (lets a peer mirror your spectrum + see your GPS/hits) and a control key (lets them tune your VFOs, change mission). Useful if you want to invite an analyst without giving them the keys to your radio.
I'd suggest A for v1, with B as a follow-on if you want it. Either way the key goes over TLS so it's not sniffable on the VLAN.
Topology — full mesh or hub & spoke?

Mesh: every node both serves and connects to others. Any operator can pull from any peer. More flexible, slightly more bookkeeping.
Hub & spoke: one operator console pulls from many sensor nodes. Simpler.
You said "control other devices", plural, in either direction implied. I'd build mesh — every Predator RF instance runs both a tiny server and a client, and you decide per-instance whether to advertise.
